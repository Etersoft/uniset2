# Design Doc: JScript Debug Visualizer

Created: 2026-04-04
Status: Draft
Author: Pavel Vainerman

## Overview

Web-based debugger/visualizer for monitoring and debugging JavaScript programs
running on UniSet2 JScript (QuickJS). Provides real-time variable monitoring,
execution trace, and historical trend charts — all through a browser.

## Goals

- **Works with any JS script** — no st2js or code generation required
- Live monitoring of inputs, outputs, local variables, and FB states
- Execution trace: which IF/CASE branches fire each cycle (optional instrumentation)
- History/trends: time-series charts for selected variables
- Zero installation on client side (browser-only)
- Minimal performance impact on JScript runtime when disabled
- Optional integration with st2js converter (`--debug` flag for trace instrumentation)

## Usage Modes

### Mode 1: Any JS script (no st2js)

```javascript
// user-written script, no code generation
load("uniset2-iec61131.js");
load("uniset2-debug.js");
uniset_debug_start(8088);

uniset_inputs = [{ name: "AI_Temp_S" }];
uniset_outputs = [{ name: "DO_Heater_C" }];

const myTimer = new TON(3000);

function uniset_on_step() {
    myTimer.update(!!in_AI_Temp_S);
    out_DO_Heater_C = myTimer.Q ? 1 : 0;
}
```

Available: live variables, FB status, history/trends, force override.
Not available: execution trace (requires `_dbg_if`/`_dbg_case` calls).

### Mode 2: st2js-generated script with --debug

```bash
python -m st2js program.st --debug -o output.js
```

Generated JS includes `_dbg_if()`/`_dbg_case()` trace instrumentation.
Available: everything from Mode 1 + execution trace.

### Mode 3: Manual trace in hand-written JS

```javascript
load("uniset2-debug.js");
uniset_debug_start(8088);

function uniset_on_step() {
    if (_dbg_if(1, in_Temperature > 80)) {  // manual trace point
        out_Alarm = 1;
    }
}
```

Selective trace for specific branches without full st2js instrumentation.

## Debug Metadata

### Problem

Through `globalThis` the debugger can see variable values, but loses semantics:
- `in_AI_Temp_S = 2350` — is it 2350 or 23.50°C (scale 100)?
- `state = 1` — is it just 1 or "Running" (enum)?
- `myTimer` — is it a TON with PT=5000? (needed for progress bar)
- Which variables belong to which PROGRAM?

### Solution: Two metadata layers

#### Layer 1: `_debug_meta` on FB classes

Each FB class exports a static metadata descriptor. The debugger reads
`instance.constructor._debug_meta` to determine how to display the instance.

```javascript
// In uniset2-iec61131.js:
TON._debug_meta = {
    type: "timer",
    display: "progress",     // render as progress bar (ET/PT)
    fields: {
        Q:  { type: "bool", label: "Output" },
        ET: { type: "time", label: "Elapsed", unit: "ms" },
        PT: { type: "time", label: "Preset", unit: "ms", readonly: true },
    }
};

CTU._debug_meta = {
    type: "counter",
    display: "gauge",        // render as gauge (CV/PV)
    fields: {
        Q:  { type: "bool", label: "Reached" },
        CV: { type: "int",  label: "Count" },
        PV: { type: "int",  label: "Preset", readonly: true },
    }
};

RS._debug_meta = {
    type: "bistable",
    display: "indicator",    // render as on/off lamp
    fields: {
        Q1: { type: "bool", label: "Output" },
    }
};
```

```javascript
// In uniset2-iec61131-codesys.js:
PID._debug_meta = {
    type: "controller",
    display: "chart",        // render as mini chart (Y over time)
    fields: {
        Y:             { type: "real",  label: "Output" },
        LIMITS_ACTIVE: { type: "bool",  label: "Saturated" },
        OVERFLOW:      { type: "bool",  label: "Overflow" },
    }
};

BLINK._debug_meta = {
    type: "signal",
    display: "indicator",
    fields: {
        OUT: { type: "bool", label: "Output" },
    }
};

HYSTERESIS._debug_meta = {
    type: "analog",
    display: "indicator",
    fields: {
        OUT: { type: "bool", label: "Output" },
    }
};
```

User-defined FBs can also provide metadata:

```javascript
class MyController {
    constructor() { this.output = 0; this.error = 0; }
    execute() { /* ... */ }
}
MyController._debug_meta = {
    type: "controller",
    display: "custom",
    fields: {
        output: { type: "real", label: "Control Output", unit: "%" },
        error:  { type: "real", label: "Error" },
    }
};
```

When `_debug_meta` is not present, the debugger falls back to listing
all enumerable properties with raw values (generic object view).

#### Layer 2: `_program_meta` for program context

Generated by st2js (`--debug` flag) or written manually. Describes the
program structure, variable types, sensor mapping, enums.

```javascript
globalThis._program_meta = {
    name: "Thermostat",
    source: "thermostat.st",
    inputs: [
        { name: "Temperature", sensor: "AI_Temp_S", type: "REAL", scale: 100, unit: "°C" },
        { name: "Enable", sensor: "DI_Enable_S", type: "BOOL" },
    ],
    outputs: [
        { name: "HeaterOn", sensor: "DO_Heater_C", type: "BOOL" },
        { name: "CycleCount", sensor: "AO_CycleCount_C", type: "INT" },
    ],
    locals: [
        { name: "state", type: "INT",
          enum: { 0: "Idle", 1: "Running", 2: "Alarm" } },
        { name: "config", type: "STRUCT",
          fields: ["setpoint", "hysteresis", "maxTemp"] },
    ],
    fb_instances: [
        { name: "onDelay", type: "TON", args: { PT: 5000 } },
        { name: "cycleCounter", type: "CTU", args: { PV: 100 } },
        { name: "startEdge", type: "R_TRIG" },
    ],
};
```

The debugger uses `_program_meta` for:
- **Scaled display**: show `23.50°C` instead of raw `2350`
- **Enum labels**: show `state: Running` instead of `state: 1`
- **Grouping**: Inputs / Outputs / Locals / FB Instances
- **FB card rendering**: progress bar for TON (ET/PT), gauge for CTU (CV/PV)
- **Unit labels**: `°C`, `%`, `ms`

### Metadata Resolution Order

The debugger resolves display information in this order:

1. **`_program_meta`** (if present) — variable type, scale, enum, unit
2. **`_debug_meta`** on FB class (if `instanceof` matches) — display type, fields
3. **Generic fallback** — raw value, typeof-based type detection

This means:
- `_program_meta` alone is sufficient for full-featured display
- `_debug_meta` alone gives smart FB visualization without program context
- Neither present — basic raw value display (always works)

### What st2js generates with --debug

```javascript
// Auto-generated by st2js --debug:

// Stubs for when uniset2-debug.js is not loaded
if (typeof _dbg_if === 'undefined') {
    globalThis._dbg_if = function(l, c) { return c; };
    globalThis._dbg_case = function(l, v) { return v; };
    globalThis._dbg_begin_cycle = function() {};
    globalThis._dbg_end_cycle = function() {};
}

// Program metadata
globalThis._program_meta = {
    name: "Thermostat",
    source: "thermostat.st",
    inputs: [ /* ... */ ],
    outputs: [ /* ... */ ],
    locals: [ /* ... */ ],
    fb_instances: [ /* ... */ ],
};

load("uniset2-iec61131.js");
// ... rest of generated code with _dbg_if()/_dbg_case() calls ...
```

### Summary: No separate debugger-per-library needed

| What | Where | Why |
|------|-------|-----|
| `_debug_meta` on FB classes | In each library file (5-10 lines per class) | Metadata next to code, never out of sync |
| `_program_meta` | Generated by st2js or written by user | Program-level context (scale, enum, grouping) |
| `uniset2-debug.js` | Single debug module | Reads both metadata layers, one module to load |
| Web UI | `uniset2-debug-ui.html` | Adapts display based on available metadata |

## Non-Goals (v1)

- Breakpoints / single-step execution (requires QuickJS runtime modification)
- Source-level mapping to original ST code (future: source maps)
- Watchpoints / conditional breakpoints
- Multi-process debugging (one visualizer per JScript process)

```
|  |          |  |  - trace collector     |    |
|  | user.js  |  |  - history ring buffer |    |
|  +----------+  +-----------+-----------+    |
|                            | JSON            |
|                +-----------v-----------+    |
|                | JHttpServer (HTTP)    |    |
|                | (via uniset.http_start)|    |
|                +-----------+-----------+    |
+----------------------------|-----------------+

                +------------v----------------+

                |  +------------------------+  |
                |  | Variables  | FB Status |  |
                |  +------------------------+  |
                |  | Exec Trace | Trends    |  |
                |  +------------------------+  |
                |  (polls GET /debug/snapshot)  |
                +------------------------------+
```

## Execution Order

The JScript engine executes callbacks in the following order each cycle:

```
timers → step_cb → uniset_on_step → httpLoop
```

1. **timers** — internal timer callbacks fire
2. **step_cb** — registered step callbacks run (force overrides applied here)
3. **uniset_on_step** — user program logic runs
4. **httpLoop** — HTTP request handlers run (snapshot collection happens here)

This means:
- Force overrides are applied as INPUT to `uniset_on_step()` (step_cb runs before it)
- Snapshots capture the state AFTER `uniset_on_step()` has completed (httpLoop runs after it)

## Components

### 1. Debug Runtime Module (`uniset2-debug.js`)

JS module loaded into JScript process. Provides the debug infrastructure.

#### API

```javascript
load("uniset2-debug.js");

// Start debug server (HTTP on given port, served via existing JHttpServer)
uniset_debug_start(8088, {
    history_depth: 1000,    // ring buffer size (cycles)
    trace_enabled: true,    // collect execution trace
});

// Manual variable registration (optional, auto-detected from globals)
uniset_debug_watch("myVar", () => myVar);
uniset_debug_watch("myTimer.Q", () => myTimer.Q);
```

All endpoints are served through the existing JHttpServer via `uniset.http_start()`.
The debug module registers HTTP handlers under the `/debug/` path prefix to avoid
conflicts with user-defined routes.

#### Internal Behavior

- Registers a `uniset.step_cb()` callback that runs BEFORE each `uniset_on_step()`:
  - Applies forced variable values (so they become inputs to the program)
- Snapshot collection happens in the HTTP handler (httpLoop), which runs AFTER `uniset_on_step()`:
  - All `in_*` and `out_*` global variables (auto-detected via globalThis scan)
  - Explicitly watched variables
  - Trace events from `_dbg_*` calls (if instrumented code)
- Stores snapshots in a fixed-size ring buffer (configurable depth)
- Ring buffer always fills regardless of whether a client is connected
  (useful for late-connect history review)
- globalThis scan only runs when a snapshot is actually requested
  (lazy evaluation, triggered by HTTP request)
- HTTP handlers process incoming requests and return JSON responses

#### Variable Auto-Detection

Scans `globalThis` for:
- `in_*` prefixed variables (inputs)
- `out_*` prefixed variables (outputs)
- Known FB instance names (from loaded `uniset2-iec61131.js` classes)
- User-registered variables via `uniset_debug_watch()`

### 2. Trace Instrumentation (st2js `--debug` flag)

When st2js generates JS with `--debug`, it inserts trace calls:

#### Without `--debug` (normal):
```javascript
if ((in_Temperature > 80.0)) {
    out_Alarm = true;
}
```

#### With `--debug`:
```javascript
if (_dbg_if(42, (in_Temperature > 80.0))) {
    out_Alarm = true;
}
```

#### Trace Functions

```javascript
// Provided by uniset2-debug.js:
function _dbg_begin_cycle() { /* reset trace buffer */ }
function _dbg_end_cycle()   { /* finalize trace for this cycle */ }

// Returns the condition value unchanged (transparent wrapper)
function _dbg_if(line, cond) {
    _trace.push({type: "if", line: line, result: !!cond});
    return cond;
}

function _dbg_case(line, value) {
    _trace.push({type: "case", line: line, value: value});
    return value;
}

// When debug.js is NOT loaded, stubs are defined:
// function _dbg_if(l, c) { return c; }
// function _dbg_case(l, v) { return v; }
```

The `--debug` flag in st2js wraps IF conditions and CASE selectors.
Performance overhead: one function call per IF/CASE per cycle (~negligible
for typical PLC programs with <100 branches).

When `uniset2-debug.js` is not loaded, the `_dbg_*` functions are
defined as transparent pass-through stubs at the top of generated JS,
so `--debug` code works without the debug module (just no trace collection).

### 3. HTTP API

All debug endpoints use the `/debug/` path prefix and are served through
the existing JHttpServer via `uniset.http_start()`. The browser polls
these endpoints at a configurable interval (default 300ms).

#### Endpoints

```
GET /debug/snapshot
  Returns the LAST collected snapshot plus a cycles_since_last_poll counter
  indicating how many cycles passed since the previous poll from this client.

  Response:
  {
      "cycle": 12345,
      "ts": 1712234567890,
      "dt_ms": 152,
      "vars": {
          "in_AI_Temp_S": 2350,
          "in_DI_Enable_S": 1,
          "out_DO_Heater_C": 1,
          "out_DO_Alarm_C": 0,
          "state": 1,
          "myTimer.Q": false,
          "myTimer.ET": 2500,
          "cycleCounter.CV": 42
      },
      "trace": [
          {"type": "if", "line": 42, "result": true},
          {"type": "case", "line": 55, "value": 1},
          {"type": "if", "line": 60, "result": false}
      ],
      "forced": ["out_DO_Heater_C"],
      "cycles_missed": 3
  }

GET /debug/history?var=in_AI_Temp_S&depth=500
  Returns ring buffer data for the specified variable (for trend charts).

  Response:
  {
      "var": "in_AI_Temp_S",
      "data": [
          [1712234567000, 2300],
          [1712234567150, 2350],
          [1712234567300, 2380]
      ]
  }

GET /debug/info
  Returns debug server metadata.

  Response:
  {
      "version": "1.0",
      "program": "Thermostat",
      "cycle_ms": 150,
      "vars_count": 15,
      "history_depth": 1000,
      "uptime_ms": 123456
  }

POST /debug/force
  Force-set a variable value (debug override).

  Request body: { "var": "out_DO_Heater_C", "value": 1 }
  Response: { "ok": true }

POST /debug/unforce
  Release a forced variable.

  Request body: { "var": "out_DO_Heater_C" }
  Response: { "ok": true }

POST /debug/config
  Change runtime settings.

  Request body: { "history_depth": 2000 }
  Response: { "ok": true }

GET /debug/ui
  Serves the HTML debug page (uniset2-debug-ui.html).
```

#### Error Response

```json
{
    "error": "unknown variable: foo"
}
```

### 4. Web UI (`uniset2-debug-ui.html`)

Single-file HTML with inline JS/CSS. Served by the debug module's HTTP handler
at `GET /debug/ui`. The UI uses `fetch()` polling to retrieve data from the
debug HTTP endpoints.

#### Connection Logic

```javascript
// Polling loop (configurable interval, default 300ms)
let pollInterval = 300;
let polling = true;

async function pollLoop() {
    while (polling) {
        try {
            const resp = await fetch('/debug/snapshot');
            const snapshot = await resp.json();
            updateVariablesPanel(snapshot.vars);
            updateTracePanel(snapshot.trace);
            updateForcedIndicators(snapshot.forced);
            updateCycleInfo(snapshot.cycle, snapshot.dt_ms, snapshot.cycles_missed);
        } catch (e) {
            showDisconnected();
        }
        await new Promise(r => setTimeout(r, pollInterval));
    }
}

// Initial info fetch
fetch('/debug/info').then(r => r.json()).then(info => {
    showConnected(info);
    pollLoop();
});
```

#### Panels

**Variables Panel** (left side):
- Table: Name | Value | Type | Changed
- Grouped: Inputs / Outputs / Locals / FB Instances
- Color highlight on value change (flash green/red)
- Click to add to trend chart
- Right-click to force value

**FB Status Panel** (top right):
- Card per FB instance:
  ```
  +-- myTimer (TON, PT=5000) --+
  |  IN: true   Q: false       |
  |  ET: 2500 / 5000  [=====  ]|
  +-----------------------------+
  ```
- Visual progress bar for ET/PT on timers
- Counter value display for CTU/CTD/CTUD

**Execution Trace Panel** (middle right):
- Current cycle's trace: list of decisions
- Format: `IF @42: true`, `CASE @55: → 1`
- Highlight: green=true, red=false
- Shows last N cycles with expand/collapse

**Trend Chart Panel** (bottom):
- Canvas-based line chart (no external dependencies)
- Multiple variables on same chart (different colors)
- Auto-scaling Y axis
- Time window: last 30s / 1m / 5m / all history
- Pause/resume, zoom, export CSV
- History data fetched via `GET /debug/history?var=...&depth=...`

**Control Bar** (top):
- Connection status: `Connected to http://host:8088/debug/`
- Cycle counter and frequency: `#12345, 150ms/cycle`
- Cycles missed since last poll indicator
- Pause/resume polling
- Settings: poll interval, history depth

#### Technology

- Vanilla JS (no framework, no build step)
- CSS Grid layout
- Canvas for charts (or SVG for simple cases)
- `fetch()` for HTTP polling (no WebSocket dependency)
- Single HTML file (<50KB) served from JScript process
- Works in any modern browser

### 5. st2js Integration

#### CLI Changes

```bash
# Normal conversion
python -m st2js program.st -o output.js

# With debug instrumentation
python -m st2js program.st --debug -o output.js
```

#### Generated Code with --debug

```javascript
// Header stubs (in case uniset2-debug.js is not loaded)
if (typeof _dbg_if === 'undefined') {
    globalThis._dbg_if = function(l, c) { return c; };
    globalThis._dbg_case = function(l, v) { return v; };
    globalThis._dbg_begin_cycle = function() {};
    globalThis._dbg_end_cycle = function() {};
}

load("uniset2-iec61131.js");

// ... sensor arrays, variables, FB instances ...

function uniset_on_step() {
    _dbg_begin_cycle();

    // IF Temperature > 80.0 THEN  (thermostat.st:42)
    if (_dbg_if(42, (in_Temperature > 80.0))) {
        out_Alarm = true;
    }

    // CASE state OF  (thermostat.st:55)
    switch (_dbg_case(55, state)) {
        case 0:
            // ...
    }

    _dbg_end_cycle();
}
```

Line numbers in `_dbg_if(42, ...)` refer to the generated JS line
(or optionally to the original ST line if source map is available).

### 6. Force Override Mechanism

The debug module maintains a `_forced` map. When a variable is forced:
- The forced value is written to the global variable in `step_cb`, BEFORE `uniset_on_step()`
- This means forced values act as INPUT to the user program
- The user program MAY override them during execution
- Forced variables are highlighted in the UI
- Force persists until explicitly released

```javascript
// Internal mechanism in uniset2-debug.js:
const _forced = new Map();  // varName -> forcedValue

// In step_cb (runs BEFORE uniset_on_step):
for (const [name, value] of _forced) {
    globalThis[name] = value;
}
```

**v1 Limitation**: Because force is applied before `uniset_on_step()`, the program
can overwrite forced output variables during execution. Force reliably works for
input variables and local variables that are read (not written) by the program.
For output forcing, a post-step mechanism would be needed (planned for v2) OR
the user must accept that force only guarantees the value at the START of the cycle.

## Performance Considerations

| Component | Overhead | When |
|-----------|----------|------|
| Variable snapshot | ~0.1ms/cycle | Only when HTTP request received (lazy) |
| Trace collection | ~0.05ms/branch | Only with `--debug` instrumented code |
| History ring buffer | ~400KB for 50 vars x 1000 cycles | 50 vars x 8 bytes x 1000 cycles (values + timestamps) |
| HTTP response | ~0.2ms/request | Only when client polls |
| Ring buffer fill | ~0.05ms/cycle | Always when debug module loaded (no client needed) |
| No debug module | 0 | `_dbg_*` stubs are no-ops |

For typical PLC programs (10-50 variables, <100 branches), total debug
overhead is <1ms per 150ms cycle (~0.7%).

### Phase 5: Connection Graph Visualization

Extract FB connection graph and visualize as interactive diagram in Web UI.

#### Data Sources for Connections

**Source 1: ST code (via st2js IR)**

FB calls with named parameters encode connections:
```
startEdge(CLK := Enable);              → Enable --> startEdge.CLK
onDelay(IN := HeaterOn, PT := T#5s);   → HeaterOn --> onDelay.IN, 5000 --> onDelay.PT
cycleCounter(CU := onDelay.Q, ...);    → onDelay.Q --> cycleCounter.CU
```

Extracted during st2js `--debug` codegen by walking IRFBCall nodes.

**Source 2: PLCopen XML FBD sections**

FBD diagrams store connections explicitly:
```xml
<block localId="1" typeName="TON">
  <inputVariables>
    <variable formalParameter="IN">
      <connectionPointIn>
        <connection refLocalId="3" />   <!-- connected to element 3 -->
      </connectionPointIn>
    </variable>
  </inputVariables>
</block>
<inVariable localId="3">
  <expression>Enable</expression>
</inVariable>
```

Extracted during PLCopen XML parsing in parser.py.

#### _program_meta.connections format

```javascript
globalThis._program_meta = {
    // ... existing fields ...
    connections: [
        { from: "Enable",     to: "startEdge",     toParam: "CLK" },
        { from: "HeaterOn",   to: "onDelay",       toParam: "IN" },
        { from: "onDelay",    to: "cycleCounter",  toParam: "CU",    fromField: "Q" },
        { from: "Reset",      to: "cycleCounter",  toParam: "RESET" },
    ]
};
```

Each connection: `from` (variable or FB instance), `fromField` (output field, optional),
`to` (FB instance), `toParam` (input parameter name).

Constants (PT, PV) are stored as FB constructor args, not as connections.

- SVG-based graph with FB blocks as rectangles, variables as circles
- Connections drawn as lines/arrows between blocks
